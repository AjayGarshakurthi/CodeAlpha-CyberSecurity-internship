💉 SQL Injection

CRITICAL SQL-01 · CWE-89 · Line 47

Login SQL Injection

Username and password are concatenated directly into the SQL query string. An attacker can input ' OR '1'='1 to bypass authentication entirely, or use UNION-based injection to dump the entire database.

query = "SELECT * FROM users WHERE username='" + username + "' AND password='"

✅ Fix: Use parameterised queries:
db.execute("SELECT * FROM users WHERE username=? AND password=?", (username, hashed))

CRITICAL SQL-02 · CWE-89 · Line 68

Search SQL Injection

The q GET parameter is concatenated into a LIKE query. Allows data exfiltration, authentication bypass, and in some configurations, file read/write via SQL.

"SELECT username, role FROM users WHERE username LIKE '%" + query + "%'"

✅ Fix: db.execute("SELECT ... WHERE username LIKE ?", (f"%{query}%",))

💻 OS Command Injection

CRITICAL CMD-01 · CWE-78 · Line 76

Command Injection via ping endpoint

The host parameter is passed directly to subprocess.check_output() with shell=True. An attacker can supply localhost; cat /etc/passwd to execute arbitrary OS commands.

subprocess.check_output("ping -c 1 " + host, shell=True, ...)

✅ Fix: Use a list with shell=False and validate input with a strict regex:
subprocess.run(["ping", "-c", "1", host], shell=False, timeout=5)

HIGH CMD-02 · CWE-78 · General

Avoid os.system()

os.system() passes commands to the shell and is inherently injection-prone. It also provides no way to capture output safely.

✅ Fix: Replace all os.system() calls with subprocess.run() using a list of arguments.

🔑 Hardcoded Credentials

CRITICAL CRED-01 · CWE-798 · Lines 18–20

Hardcoded Passwords & Secret Key

Admin credentials and the Flask secret key are hardcoded in source. Anyone with repository access can immediately compromise the application and forge session cookies.

app.secret_key = "secret123" / ADMIN_PASS = "password123"

✅ Fix: Load all secrets from environment variables:
app.secret_key = os.environ["FLASK_SECRET_KEY"]

HIGH CRED-02 · CWE-798 · Line 18

Weak / Guessable Secret Key

Even if moved to an env var, "secret123" is trivially guessable. A weak secret key allows attackers to forge session cookies and impersonate any user.

✅ Fix: Generate a cryptographically random key: secrets.token_hex(32)

🔐 Weak Cryptography

HIGH CRYPT-01 · CWE-327 · Lines 30, 31, 107

MD5 Password Hashing (No Salt)

MD5 is cryptographically broken. Unsalted MD5 hashes are trivially cracked using rainbow tables. The entire user database can be compromised in seconds with freely available tools.

hashlib.md5(b"password123").hexdigest()

✅ Fix: Use bcrypt or argon2 with a random salt:
bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

📦 Insecure Deserialization

CRITICAL DESER-01 · CWE-502 · Line 97

Arbitrary Code Execution via pickle

pickle.loads() on attacker-controlled hex data allows Remote Code Execution (RCE). An attacker crafts a malicious pickle payload that executes any OS command when deserialized.

obj = pickle.loads(bytes.fromhex(data)) # data from request.args

✅ Fix: Never deserialize untrusted data with pickle. Use JSON with a strict schema validator (e.g., pydantic).

🌐 XSS & Template Injection

HIGH TMPL-01 · CWE-94 · Lines 55–65

Server-Side Template Injection (SSTI)

The login error message is concatenated into a template string passed to render_template_string(). An attacker can inject Jinja2 expressions like {{7*7}} or {{config}} to read secrets or achieve RCE.

template = "..." + error + "..." ; render_template_string(template)

✅ Fix: Use static .html template files with render_template() and pass data as context variables — never build templates from user input.

HIGH XSS-01 · CWE-79 · Lines 68–73

Reflected XSS in Search

The search query is reflected directly into the HTML response without escaping. An attacker can craft a URL with <script>alert(document.cookie)</script> to steal session cookies.

output = f"<h2>Results for: {query}</h2>"

✅ Fix: Use Jinja2 templates (auto-escaped) or markupsafe.escape(query) before rendering.

📁 Path Traversal

HIGH PATH-01 · CWE-22 · Lines 86–88, 100–101

Path Traversal in File Upload & Export

File paths are constructed from user-supplied filenames without sanitisation. An attacker can supply ../../etc/passwd to read or overwrite arbitrary files on the server.

f.save(os.path.join("/var/uploads", f.filename))

✅ Fix: Use werkzeug.utils.secure_filename() and verify the resolved path stays within the intended directory using os.path.realpath().

MEDIUM PATH-02 · CWE-22 · Line 100

Unrestricted File Upload

No file type or size validation. An attacker can upload a .py or .php web shell and potentially execute it, or upload a multi-GB file to exhaust disk space.

✅ Fix: Whitelist allowed extensions, validate MIME type, enforce MAX_CONTENT_LENGTH in Flask config.

🔒 Broken Access Control

HIGH AUTH-01 · CWE-284 · Line 110

Role Check from Client Cookie

The admin panel reads the role value from a plain (unsigned) cookie. Any user can open DevTools, set role=admin, and gain full admin access.

role = request.cookies.get("role", "user")

✅ Fix: Store roles in the server-side session (Flask's signed session cookie) or in the database. Never trust client-supplied role values.

HIGH AUTH-02 · CWE-284 · Line 62

Missing Authentication on Dashboard

The /dashboard route has no authentication check. Any unauthenticated user can access it directly.

✅ Fix: Add a @login_required decorator to all protected routes.

MEDIUM AUTH-03 · CWE-639 · Line 80

Insecure Direct Object Reference (IDOR)

The /profile/<username> route returns any user's profile without checking if the requester owns that profile. An attacker can enumerate all usernames.

✅ Fix: Verify session["user_id"] == requested_user_id or restrict to admin role.

⚙️ Insecure Configuration

HIGH CFG-01 · CWE-94 · Line 130

Debug Mode Enabled in Production

app.run(debug=True) exposes an interactive Werkzeug debugger with a PIN-protected console. If the PIN is bypassed or leaked, attackers get a Python shell on the server.

app.run(debug=True, host="0.0.0.0", port=5000)

✅ Fix: debug=False always in production. Control via FLASK_DEBUG env var.

MEDIUM CFG-02 · CWE-605 · Line 130

Binding to All Interfaces

host="0.0.0.0" exposes the development server on all network interfaces, making it reachable from outside the local machine.

✅ Fix: Bind to 127.0.0.1 in development. Use a production WSGI server (gunicorn + nginx) in deployment.

MEDIUM CFG-03 · CWE-215 · Line 115

Debug Endpoint Exposes Environment Variables

The /debug route dumps all environment variables including secrets, database URLs, and API keys to any visitor.

return str(dict(os.environ))

✅ Fix: Remove all debug/diagnostic endpoints before deployment.

MEDIUM CFG-04 · CWE-601 · Line 120

Open Redirect

The /redirect endpoint accepts any URL from the query string without validation. Attackers use this to craft convincing phishing links that appear to originate from a trusted domain.

return redirect(request.args.get("url", "/"))

✅ Fix: Reject URLs with external schemes. Only allow relative paths or a strict allowlist of trusted domains.

🛡️ CSRF

MEDIUM CSRF-01 · CWE-352 · Line 123

Missing CSRF Protection on Password Reset

The /reset_password POST endpoint has no CSRF token. An attacker can host a malicious page that silently submits a form to change the victim's password when they visit it.

✅ Fix: Use Flask-WTF's CSRFProtect and include {{ form.hidden_tag() }} in all forms.
